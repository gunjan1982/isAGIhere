Overview
This plan moves isagihere.wiki completely off Replit — including the Express API, React/Vite frontend, PostgreSQL database, background jobs, and all secrets — onto Google Cloud Platform (GCP). After cutover, no runtime dependency on Replit infrastructure remains.
The migration also serves as the v2 deploy opportunity: all three pending v2 features (YouTube Interview Monitor, AI YouTube Channels Directory, MY AI Journey) will be deployed fresh on GCP with their DB migrations applied, eliminating the backlog of Replit-pending work.

PHASE 1: Database Migration   Half day

1.1  Create Cloud SQL Instance
gcloud sql instances create isagihere-db \
  --database-version=POSTGRES_16 \
  --tier=db-g1-small \
  --region=us-central1 \
  --storage-type=SSD \
  --storage-size=10GB \
  --backup-start-time=03:00 \
  --no-assign-ip \
  --network=default

# Create the database and user
gcloud sql databases create isagihere --instance=isagihere-db
gcloud sql users create isagihere_user \
  --instance=isagihere-db \
  --password=<strong-password>

1.2  Export Database from Replit
Run from Replit Shell. The export will capture all v1 production data (people, sources, communities, feed_items, comments, submissions).
# In Replit Shell
pg_dump $DATABASE_URL \
  --format=custom \
  --no-acl \
  --no-owner \
  -f isagihere_export.dump

# Verify dump is non-empty
ls -lh isagihere_export.dump

1.3  Import to Cloud SQL
# Upload dump to GCS bucket (create if needed)
gsutil mb gs://isagihere-migration
gsutil cp isagihere_export.dump gs://isagihere-migration/

# Import via Cloud SQL
gcloud sql import pgdump isagihere-db \
  gs://isagihere-migration/isagihere_export.dump \
  --database=isagihere

PHASE 2: Containerise & Deploy API to Cloud Run   1 day

2.1  Write the Dockerfile
Add this file at artifacts/api-server/Dockerfile. The build uses esbuild to produce a single CJS bundle, then runs it on a minimal Node 24 Alpine image:
# artifacts/api-server/Dockerfile
FROM node:24-alpine AS builder
WORKDIR /app
COPY . .
RUN npm install -g pnpm && pnpm install --frozen-lockfile
RUN pnpm --filter @workspace/api-server run build

FROM node:24-alpine AS runtime
WORKDIR /app
COPY --from=builder /app/artifacts/api-server/dist ./dist
COPY --from=builder /app/artifacts/api-server/package.json .
RUN npm install -g pnpm && pnpm install --prod --frozen-lockfile
EXPOSE 8080
CMD ["node", "dist/index.cjs"]

2.2  Build & Push Image
# From repo root
docker build -f artifacts/api-server/Dockerfile \
  -t us-central1-docker.pkg.dev/isagihere-prod/isagihere/api:latest .

docker push us-central1-docker.pkg.dev/isagihere-prod/isagihere/api:latest

2.3  Create Service Account for Cloud Run
# Service account for the API
gcloud iam service-accounts create isagihere-api \
  --display-name='isagihere API service account'

# Grant access to Cloud SQL
gcloud projects add-iam-policy-binding isagihere-prod \
  --member='serviceAccount:isagihere-api@isagihere-prod.iam.gserviceaccount.com' \
  --role='roles/cloudsql.client'

# Grant access to Secret Manager
gcloud projects add-iam-policy-binding isagihere-prod \
  --member='serviceAccount:isagihere-api@isagihere-prod.iam.gserviceaccount.com' \
  --role='roles/secretmanager.secretAccessor'

2.4  Deploy to Cloud Run
gcloud run deploy isagihere-api \
  --image=us-central1-docker.pkg.dev/isagihere-prod/isagihere/api:latest \
  --region=us-central1 \
  --platform=managed \
  --allow-unauthenticated \
  --service-account=isagihere-api@isagihere-prod.iam.gserviceaccount.com \
  --add-cloudsql-instances=isagihere-prod:us-central1:isagihere-db \
  --set-secrets=DATABASE_URL=DATABASE_URL:latest \
  --set-secrets=CLERK_SECRET_KEY=CLERK_SECRET_KEY:latest \
  --set-secrets=ANTHROPIC_API_KEY=ANTHROPIC_API_KEY:latest \
  --set-secrets=RESEND_API_KEY=RESEND_API_KEY:latest \
  --set-env-vars=NODE_ENV=production \
  --min-instances=1 \
  --max-instances=10 \
  --memory=512Mi \
  --cpu=1
Set --min-instances=1 to avoid cold-start latency on the RSS/YouTube schedulers. Adjust up if needed after observing traffic patterns.

2.5  Verify API Health
# Cloud Run will give you a URL like:
# https://isagihere-api-xxxx-uc.a.run.app

curl https://isagihere-api-xxxx-uc.a.run.app/api/healthz
# Expected: { "status": "ok" }

curl https://isagihere-api-xxxx-uc.a.run.app/api/people | head -c 200
# Expected: JSON array of people

PHASE 4: Scheduled Jobs via Cloud Scheduler   2 hours

The current code runs jobs via setInterval() inside the Express process. On Cloud Run this is fine as long as min-instances=1, but it is better practice to extract them into dedicated HTTP endpoints triggered by Cloud Scheduler. This is more reliable, observable, and survives restarts.
4.1  Expose Scheduler Endpoints in the API
Add two new internal routes to artifacts/api-server/src/routes/scheduler.ts (restrict to scheduler service account only):
// artifacts/api-server/src/routes/scheduler.ts
router.post('/api/internal/refresh-feeds', verifyScheduler, async (req, res) => {
  await refreshFeeds();
  res.json({ ok: true });
});

router.post('/api/internal/fetch-interviews', verifyScheduler, async (req, res) => {
  await startInterviewScheduler();  // single pass
  res.json({ ok: true });
});

// verifyScheduler middleware: checks OIDC token from Cloud Scheduler
// Use google-auth-library to validate the bearer token

4.2  Create Cloud Scheduler Jobs
# RSS feed refresh — every 30 minutes
gcloud scheduler jobs create http refresh-feeds \
  --location=us-central1 \
  --schedule='*/30 * * * *' \
  --uri='https://isagihere-api-xxxx-uc.a.run.app/api/internal/refresh-feeds' \
  --http-method=POST \
  --oidc-service-account-email=isagihere-api@isagihere-prod.iam.gserviceaccount.com \
  --oidc-token-audience='https://isagihere-api-xxxx-uc.a.run.app'

# YouTube interview fetcher — every hour
gcloud scheduler jobs create http fetch-interviews \
  --location=us-central1 \
  --schedule='0 * * * *' \
  --uri='https://isagihere-api-xxxx-uc.a.run.app/api/internal/fetch-interviews' \
  --http-method=POST \
  --oidc-service-account-email=isagihere-api@isagihere-prod.iam.gserviceaccount.com \
  --oidc-token-audience='https://isagihere-api-xxxx-uc.a.run.app'

4.3  Remove In-Process setInterval() from API
Once Cloud Scheduler is verified, remove the startup calls to refreshFeeds() and startInterviewScheduler() from artifacts/api-server/src/index.ts. The scheduler is now the source of truth for cadence.

Rollback Plan
If any phase fails and the issue cannot be resolved within 30 minutes during the cutover window:
	•	Revert DNS A records for isagihere.wiki back to the Replit autoscale IP. Because TTL was reduced to 60s in Phase 5.1, propagation takes < 2 minutes.
	•	Replit remains running in read-write mode throughout the migration — no writes were disabled.
	•	The Cloud SQL database is not connected to Replit's data path, so reverting DNS does not cause data loss on either side.
	•	Diagnose the GCP issue, fix, then re-execute the cutover sequence from Step 5.2.

Critical: do NOT shut down the Replit deployment until you have confirmed 7+ days of stable operation on GCP.


Appendix — Key GCP Concepts
Term
What it does in this stack
Cloud Run
Runs your containerised Express server. Scales to zero when idle, scales out under load. Replaces Replit autoscale.
Firebase Hosting
Serves the compiled React/Vite dist/ as a CDN-backed static site. Free SSL, global edge caching. Replaces Replit's static file serving.
Cloud SQL
Fully managed PostgreSQL. Automatic backups, point-in-time recovery. Replaces Replit's managed DB.
Secret Manager
Stores API keys/DB URLs as versioned secrets. Cloud Run mounts them as env vars at startup. Replaces Replit Secrets.
Artifact Registry
Docker image registry. Stores every build of the API container. Cloud Run pulls from here on each deploy.
Cloud Scheduler
Cron-as-a-service. Fires authenticated HTTP POST requests to Cloud Run endpoints on a schedule. Replaces in-process setInterval().
Cloud Build
Runs your Docker build in GCP. Optional — GitHub Actions can also build locally and push.
Cloud Run domain mappings
Maps a custom subdomain (api.isagihere.wiki) to a Cloud Run service with automatic SSL.
